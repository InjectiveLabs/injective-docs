# 전략 개발 가이드

## 구성 가이드

Injective Trader는 YAML 구성 파일을 사용하여 동작, 컴포넌트 및 전략 파라미터를 정의합니다.

집중해야 할 가장 중요한 구성 섹션은 다음과 같습니다:

* `LogLevel`
* `Components` 섹션 아래의 `Initializer`에서 `Network` 및 `MarketTickers`
* `Strategies` 섹션

다음은 구성 구조에 대한 상세 설명입니다:

### 최상위 파라미터

```yaml
Exchange: Helix                # 사용할 거래소
LogLevel: INFO                 # 로깅 레벨 (DEBUG, INFO, WARNING, ERROR)
```

### Components 섹션

`Components` 섹션은 프레임워크 컴포넌트를 구성합니다:

```yaml
Components:
  # 체인 초기화 및 시장 설정
  Initializer:
    Network: mainnet           # 연결할 네트워크 (mainnet 또는 testnet)
    MarketTickers:             # 추적할 시장 티커 (ID로 변환됨)
      - INJ/USDT PERP
      - ETH/USDT
    BotName: MyBot

  # 체인 리스닝 구성
  ChainListener:
    ReconnectionDelay: 5       # 재연결 시도 전 대기 시간(초)
    LargeGapThreshold: 50      # 오더북 스냅샷 요청을 위한 시퀀스 갭 임계값

  # 트랜잭션 브로드캐스트 구성
  MessageBroadcaster:
    ErrorCodesJson: config/error_codes.json   # tx 검증을 위한 오류 코드 조회
    GranteePool:               # authz 트랜잭션 모드용
      MaxPendingTxs: 5         # grantee당 최대 대기 트랜잭션
      ErrorThreshold: 3        # grantee 차단 전 연속 오류 수
      BlockDuration: 300       # 오류 후 grantee 차단 시간(초)
      RotationInterval: 1      # grantee 로테이션 간격(초)
    RefreshInterval: 300       # 권한 부여 새로고침 확인 간격(초)
    Batch:                     # 트랜잭션 배치 설정
      MaxBatchSize: 15         # 배치당 최대 메시지 수
      MinBatchSize: 3          # 즉시 전송을 트리거하는 최소 메시지 수
      MaxGasLimit: 5000000     # 배치당 최대 가스
      MaxBatchDelay: 0.5       # 배치 완료까지 최대 대기 시간(초)
```

<Callout icon="info" color="#07C1FF" iconType="regular">
**참고**: 대부분의 사용자는 `Network`만 설정하고 `MarketTickers`에 리스닝하려는 모든 시장을 포함하면 됩니다.
이러한 고급 컴포넌트 설정을 수정할 필요가 없습니다. 기본값은 대부분의 사용 사례에서 잘 작동합니다.
</Callout>

### Strategies 섹션

`Strategies` 섹션은 각 트레이딩 전략을 정의합니다:

```yaml
Strategies:
  SimpleStrategy:                                    # 전략 식별자 (사용자 선택)
    # 필수 파라미터
    Name: "SimpleStrategy"                           # 전략 이름 (로그에 사용)
    Class: "SimpleStrategy"                          # [필수] 인스턴스화할 Python 클래스 이름
    MarketIds:                                       # [필수] 거래할 시장
      - "0x9b9980167ecc3645ff1a5517886652d94a0825e54a77d2057cbbe3ebee015963"  # INJ/USDT PERP
    AccountAddresses:                                # [필수] 사용할 계정
      - "inj1youractualaccount..."                   # (env의 개인 키와 일치해야 함)
    TradingAccount: "inj1youractualaccount..."       # [필수] 주문 배치용 계정 (env의 개인 키와 일치해야 함)

    # 선택적 파라미터
    FeeRecipient: "inj1feerecipient..."   # 거래 수수료를 받을 주소 (해당되는 경우)
    CIDPrefix: "simple_strat"              # 클라이언트 주문 ID 접두사
    SubaccountIds: ["0x123..."]            # 사용할 특정 서브계정 (그렇지 않으면 모든 사용 가능)

    # 리스크 관리 구성 [선택사항]
    # 특정 리스크 모델이 없으면 포함하지 않아도 됩니다
    Risk: "BasicRiskModel"                 # 적용할 리스크 모델 (리스크 관리 사용 시)
    RiskConfig:                            # 리스크 임계값
      DrawdownWarning: 0.1                 # 10% 드로다운 경고 임계값
      DrawdownCritical: 0.2                # 20% 드로다운 위험 임계값
      MarginWarning: 0.7                   # 70% 마진 사용 경고
      MarginCritical: 0.8                  # 80% 마진 사용 위험
```

**필수 전략 파라미터:**

* `Class`: Python 클래스 이름과 정확히 일치해야 함
* `MarketIds`: 이 전략에서 거래할 시장 ID 목록 (hex 형식 사용)
* `AccountAddresses`: 이 전략에서 거래에 사용할 계정 목록
* `TradingAccount`: 주문 실행에 사용되는 계정 (`AccountAddresses`에 있어야 함)

**권장 파라미터:**

* `CIDPrefix`: 클라이언트 주문 ID 접두사 (주문 식별에 도움)
* `Name`: 로그 및 모니터링을 위한 사람이 읽을 수 있는 이름

**사용자 정의 파라미터:**

* 전략에 필요한 모든 사용자 정의 파라미터를 추가할 수 있음
* 전략 이름 아래의 모든 파라미터는 `self.config`에서 사용 가능
* 명확성을 위해 관련 파라미터를 `Parameters` 섹션 아래에 그룹화

### 트레이딩 모드 구성

프레임워크는 두 가지 트레이딩 모드를 지원합니다:

#### 직접 실행 모드

```yaml
Strategies:
  SimpleStrategy:
    # 기타 파라미터...
    TradingAccount: "inj1youraccount..."   # 트랜잭션에 서명하고 브로드캐스트할 계정
```

#### 권한 부여 (Authz) 모드

```yaml
Strategies:
  SimpleStrategy:
    # 기타 파라미터...
    Granter: "inj1granteraccount..."   # 거래 실행 권한을 부여하는 계정
    Grantees:                          # granter를 대신하여 거래를 실행할 수 있는 계정
      - "inj1grantee1..."
      - "inj1grantee2..."
```

<Callout icon="info" color="#07C1FF" iconType="regular">
**참고**: 직접 실행을 위해 `TradingAccount`를 지정하거나 권한 부여 모드를 위해 `Granter`와 `Grantees`를 지정해야 합니다.
프레임워크는 초기화 중에 이 요구 사항을 적용합니다.
</Callout>

### RetryConfig 섹션

`RetryConfig` 섹션은 네트워크 작업에 대한 재시도 동작을 제어합니다:

```yaml
RetryConfig:
  # 전역 재시도 설정
  DefaultRetry:
    max_attempts: 3            # 최대 재시도 횟수
    base_delay: 1.0            # 재시도 간 기본 지연(초)
    max_delay: 32.0            # 최대 지연 상한(초)
    jitter: true               # 지연에 랜덤성 추가
    timeout: 30.0              # 작업 타임아웃(초)
    error_threshold: 10        # 회로 차단 전 오류 수
    error_window: 60           # 오류 카운팅 윈도우(초)

  # 컴포넌트별 재시도 설정 (기본값 재정의)
  ChainListener:
    max_attempts: 5            # 체인 리스너에 대한 더 많은 재시도
    base_delay: 2.0
    max_delay: 45.0
  MessageBroadcaster:
    max_attempts: 3
    base_delay: 1.0
    max_delay: 30.0
```

<Callout icon="info" color="#07C1FF" iconType="regular">
**참고**: RetryConfig는 합리적인 기본값을 가지고 있으며 특정 연결 문제가 없는 한 일반적으로 사용자 정의가 필요하지 않습니다.
</Callout>

## 전략 개발 가이드

Injective Trader의 전략은 `Strategy` 기본 클래스를 기반으로 일관된 구조를 따릅니다. 이 섹션에서는 효과적인 전략을 구축하는 방법을 설명합니다.

### Strategy 클래스 구조

전략 클래스는 기본 `Strategy` 클래스를 상속합니다:

```python
from src.core.strategy import Strategy, StrategyResult
from src.utils.enums import UpdateType, Side

class SimpleStrategy(Strategy):
    def __init__(self, logger, config):
		"""
        logger와 구성으로 전략을 초기화합니다.

        Args:
            logger: 전략 로깅을 위한 Logger 인스턴스
            config: 전략 구성 딕셔너리
                필수 키:
                - MarketIds: 거래할 시장 ID 목록
                - AccountAddresses: 사용할 계정 주소 목록
                선택 키:
                - Name: 전략 이름
                - Parameters: 전략별 파라미터
                - RiskConfig: 리스크 관리 파라미터
        """
        super().__init__(logger, config)

    def on_initialize(self, accounts, markets):
        """
        전략별 상태와 파라미터를 초기화합니다.
        전략이 업데이트 처리를 시작하기 전에 한 번 호출됩니다.

        Args:
            accounts: account_address -> Account 딕셔너리
            markets: market_id -> Market 딕셔너리
        """
        pass

    async def _execute_strategy(self, update_type, processed_data):
        """
        전략별 실행 로직.

       Args:
          update_type: 처리 중인 업데이트 유형
          processed_data: 핸들러 처리 후 업데이트별 데이터 딕셔너리
              공통 필드:
              - market_id: 시장 식별자
              - account_address: 계정 주소 (계정 업데이트용)
              - subaccount_id: 서브계정 식별자 (포지션/거래 업데이트용)

        Returns:
            StrategyResult:
            - orders
            - cancellations
            - margin updates
        """
        pass
```

#### Strategy 생성자 (`__init__`)

전략 클래스는 부모 클래스 생성자를 호출하는 생성자를 포함할 수 있습니다:

```python
def __init__(self, logger, config):
    super().__init__(logger, config)

    # 시장이나 계정이 필요 없는 사용자 정의 초기화
    self.custom_parameter = 42

    # 선택사항: 사용자 정의 핸들러 재정의
    self.handlers[UpdateType.OnOrderbook] = MyCustomOrderbookHandler(
        self.logger, self.config, self.metrics
    )

    # 선택사항: 사용자 정의 성과 지표 재정의
    self.my_metrics = MyCustomPerformanceMetrics(self.logger)
```

기본 클래스 생성자는 다음을 처리합니다:

1. 파라미터 검증 및 추출
2. 표준 메트릭 및 핸들러 설정
3. 상태 추적 컨테이너 초기화
4. 트레이딩 모드 설정 (direct 또는 authz)

<Callout icon="info" color="#07C1FF" iconType="regular">
**중요**: `__init__` 메서드는 시장 데이터나 계정 정보에 접근할 수 없습니다.
해당 리소스가 필요한 작업은 `on_initialize`를 사용하세요.
</Callout>

기본 `__init__`에서 제공하는 사용 가능한 속성:

| **속성**             | **설명**                                       | **소스**                                                             | **필수**   |
| ------------------------ | ----------------------------------------------------- | ---------------------------------------------------------------------- | -------------- |
| `self.name`              | 로그에 사용되는 전략 이름                             | config `Name`에서                                                     | YES            |
| `self.logger`            | Logger 인스턴스                                       | 전략별 로깅용                                          | YES            |
| `self.config`            | 완전한 전략 구성                       | config의 `Strategies` 섹션 아래 해당 전략 하위 섹션 | YES            |
| `self.market_ids`        | 이 전략에 관심 있는 시장 ID 목록        | config `MarketIds`에서                                                | YES            |
| `self.account_addresses` | 이 전략에 관심 있는 계정 주소 목록 | config `AccountAddresses`에서                                         | YES            |
| `self.subaccount_ids`    | 서브계정 ID 목록                                | config `SubaccountIds`에서                                            | NO             |
| `self.markets`           | market_id → `Market` 객체 딕셔너리           | 초기화 중 채워짐                                        | NO             |
| `self.accounts`          | account_address → `Account` 객체 딕셔너리    | 초기화 중 채워짐                                        | NO             |
| `self.trading_mode`      | "direct" 또는 "authz"                                   | config 기반                                                        | YES            |
| `self.fee_recipient`     | 수수료 수신자 주소                                 | config `FeeRecipient`에서                                             | NO             |
| `self.cid_prefix`        | 클라이언트 주문 ID 접두사                                | config `CIDPrefix`에서                                                | NO             |
| `self.metrics`           | 성과 추적                                  | 메트릭 및 알림 기록용                                       | YES [DEFAULT] |
| `self.handlers`          | 이벤트 핸들러 딕셔너리                             | UpdateType → Handler 객체                                           | YES [DEFAULT] |

#### 초기화 메서드 (`on_initialize`)

`on_initialize` 메서드는 프레임워크 시작 중 시장과 계정이 로드된 후 한 번 호출됩니다.

**목적**: 전략 상태와 파라미터 초기화 **파라미터**:

* `accounts`: account_address → `Account` 객체 딕셔너리
* `markets`: market_id → `Market` 객체 딕셔너리

**반환값**: [선택사항] 초기 주문이 있는 `StrategyResult`

```python
def on_initialize(self, accounts, markets):
    # 이제 모든 시장 및 계정 데이터에 접근할 수 있음

    # 예시: 시장 메타데이터 접근
    for market_id in self.market_ids:
        market = markets[market_id]
        self.logger.info(f"Market {market_id} tick sizes: "
                         f"price={market.min_price_tick}, "
                         f"quantity={market.min_quantity_tick}")

    # 예시: 시장 정보가 필요한 파라미터 초기화
    self.avg_prices = {
        market_id: markets[market_id].orderbook.tob()[0]
        for market_id in self.market_ids
        if markets[market_id].orderbook.tob()[0]
    }

    # 예시: 초기 주문 배치
    if self.config.get("PlaceInitialOrders", False):
        result = StrategyResult()
        # 초기 주문 추가...
        return result

    return None  # 초기 주문 없음
```

이 메서드는 전략 초기화 시퀀스의 일부입니다:

1. 프레임워크가 이 전략에 필요한 시장과 계정 로드
2. 로드된 데이터로 `on_initialize` 메서드 호출
3. 반환된 주문은 즉시 제출됨
4. 전략이 실행 상태로 전환

<Callout icon="info" color="#07C1FF" iconType="regular">
**팁**: 시장 또는 계정 데이터가 필요한 파라미터 초기화와 전략에 필요한 초기 주문 배치에 `on_initialize`를 사용하세요.
`Account`와 `Market`의 데이터 구조 정보는 아래를 참조하세요.
</Callout>

#### 전략 로직 (`_execute_strategy`) 메서드

`_execute_strategy` 메서드는 "전략 실행 (`execute`) 메서드"의 일부입니다. 기본 클래스 `execute` 메서드는 전체 실행 흐름을 처리합니다:

1. **초기화 확인**: 필요시 전략 초기화
2. **상태 업데이트**: 전략의 계정 및 시장 참조 업데이트
3. **데이터 처리**: 적절한 핸들러를 통해 원시 업데이트 데이터 처리
4. **전략 실행**: 처리된 데이터로 `_execute_strategy` 메서드 호출
5. **주문 보강**: 주문에 기본값 추가 (수수료 수신자, 클라이언트 ID)

이 메서드를 재정의할 필요는 거의 없습니다. 대신 사용자 정의 트레이딩 로직이 들어가는 `_execute_strategy` 구현에 집중하세요:

**목적**: 시장 데이터 분석 및 트레이딩 신호 생성 **파라미터**:

* `update_type`: 처리 중인 업데이트 유형
* `processed_data`: 관련 필드가 있는 핸들러 처리 데이터 딕셔너리

**반환값**: 주문/취소가 있는 `StrategyResult` 또는 `None`

```python
async def _execute_strategy(self, update_type, processed_data):
    # 오더북 업데이트에만 응답
    if update_type != UpdateType.OnOrderbook:
        return None

    # 시장 데이터 가져오기
    market_id = processed_data["market_id"]
    market = self.markets[market_id]

    # 현재 가격 가져오기
    bid, ask = market.orderbook.tob()
    if not bid or not ask:
        self.logger.warning(f"Incomplete orderbook for {market_id}")
        return None

    # 전략 로직 구현
    spread = (ask - bid) / bid
    if spread < self.min_spread_threshold:
        self.logger.info(f"Spread too narrow: {spread:.2%}")
        return None

    # 주문을 위한 서브계정 가져오기
    subaccount_id = self.config.get("SubaccountIds", [""])[0]
    if not subaccount_id:
        return None

    # 한도 초과를 피하기 위해 현재 포지션 확인
    position = self.get_position(subaccount_id, market_id)
    current_size = Decimal("0")
    if position:
        current_size = position.get("quantity", Decimal("0"))

    # 주문 파라미터 결정
    result = StrategyResult()

    # 새 주문 생성
    if current_size < self.max_position:
        buy_order = Order(
            market_id=market_id,
            subaccount_id=subaccount_id,
            order_side=Side.BUY,
            price=bid,
            quantity=self.order_size,
            market_type=market.market_type # v0.5.1부터 필수 필드
        )
        result.orders.append(buy_order)
        self.logger.info(f"Creating BUY order at {bid}: {self.order_size}")

    # 필요시 기존 주문 취소
    for order_hash in self.active_orders:
        result.cancellations.append({
            "market_id": market_id,
            "subaccount_id": subaccount_id,
            "order_hash": order_hash
        })

    return result
```

`_execute_strategy`에서 다음을 수행할 수 있습니다:

* 업데이트 유형별로 필터링하여 특정 이벤트 처리
* 현재 시장 데이터 및 계정 상태 접근
* 주문 배치 전 기존 포지션 확인
* 시장 조건에 따른 사용자 정의 트레이딩 로직 구현
* 새 주문 생성 및 기존 주문 취소
* 파생상품 시장의 포지션 마진 업데이트
* 모니터링 및 디버깅을 위한 전략 결정 로깅

프레임워크는 반환된 `StrategyResult`를 기반으로 트랜잭션 생성, 시뮬레이션 및 브로드캐스트와 같은 실행 세부 사항을 처리합니다.

   * 전략별 값에 `Parameters` 섹션 사용
   * 예상 파라미터 문서화

### 사용자 정의 핸들러

프레임워크는 데이터를 전략에 전달하기 전에 전문 핸들러를 통해 업데이트를 처리합니다. 데이터 처리에 대한 더 많은 제어를 위해 사용자 정의 핸들러를 만들 수 있습니다.

#### Handler 기본 클래스

모든 핸들러는 `UpdateHandler` 기본 클래스를 상속합니다:

```python
class UpdateHandler(ABC):
    def __init__(self, logger, config, metrics):
        self.logger = logger
        self.config = config
        self.metrics = metrics

    async def process(self, **update_data) -> Dict:
        """업데이트 데이터를 처리하고 처리된 결과 반환"""
        try:
            return await self._process_update(**update_data)
        except Exception as e:
            self.logger.error(f"Error processing update: {e}")
            return None

    @abstractmethod
    async def _process_update(self, **kwargs) -> Dict:
        """특정 업데이트 유형 처리. 서브클래스에서 구현해야 함."""
        pass
```

#### 사용자 정의 핸들러 만들기

사용자 정의 핸들러를 만들려면:

1. 적절한 핸들러 기본 클래스 상속
2. `_process_update` 메서드 재정의
3. 전략 생성자에서 핸들러 등록

```python
from src.core.handlers import OrderbookHandler

class MyCustomOrderbookHandler(OrderbookHandler):
    async def _process_update(self, **update_data):
        # 기본 처리 데이터 가져오기
        processed_data = await super()._process_update(**update_data)
        if not processed_data:
            return None

        # 사용자 정의 메트릭 추가
        market = update_data.get("market")
        bid, ask = market.orderbook.tob()

        if bid and ask:
            # 사용자 정의 메트릭 계산
            spread = ask - bid
            spread_pct = (ask - bid) / bid

            # 처리된 데이터에 추가
            processed_data["spread"] = spread
            processed_data["spread_pct"] = spread_pct

            # 메트릭 시스템에 기록
            self.metrics.add_custom_metric(f"{market.market_id}_spread", spread_pct)

        return processed_data
```

#### 사용자 정의 핸들러 등록

전략 생성자에서 사용자 정의 핸들러를 등록합니다:

```python
def __init__(self, logger, config):
    super().__init__(logger, config)

    # 기본 핸들러를 사용자 정의 구현으로 교체
    self.handlers[UpdateType.OnOrderbook] = MyCustomOrderbookHandler(
        self.logger, self.config, self.metrics
    )
    self.handlers[UpdateType.OnPosition] = MyCustomPositionHandler(
        self.logger, self.config, self.metrics
    )
```

### OrderStatus (Enum)

| **값**        | **설명**           |
| ---------------- | ------------------------- |
| `BOOKED`         | 주문 수락 및 활성 |
| `PARTIAL_FILLED` | 부분 체결          |
| `FILLED`         | 완전 체결         |
| `CANCELLED`      | 사용자가 취소           |
| `EXPIRED`        | 만료됨 (시간 기반)      |

### Side (Enum)

| **값** | **설명** |
| --------- | --------------- |
| `BUY`     | 매수 주문       |
| `SELL`    | 매도 주문      |

### MarketType (Enum)

| **값**    | **설명**       |
| ------------ | --------------------- |
